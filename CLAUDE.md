# Casinohub — アミューズメントカジノ向けトーナメント管理SaaS

## プロジェクト概要

アミューズメントカジノ（風営法の遊技場営業許可で運営される、換金なしのエンタメカジノ）向けのトーナメント管理SaaS。

### ターゲット
- トーナメントを定期開催しているポーカールーム
- アミューズメントカジノ店舗
- カジノバー・カジノイベント運営会社

### 解決する課題（メイン: トーナメント運営）
- トーナメントの参加者登録・チェックイン・レイトレジスト対応
- ブラインドストラクチャー管理・タイマー
- テーブル割当・テーブルブレイク時の再配置
- 順位集計・賞金/ポイント配分の自動計算
- リーダーボード・シーズン管理

### 収益モデル
- トーナメント参加者課金: ¥50/参加者〜
- チップ管理機能は提供しない（トーナメント管理のみでマネタイズ）

### 海外参考サービス
- LetsPoker（€1/プレイヤー/日、27カ国展開）
- Blind Valet（累計300万人以上）
- PokerAtlas TableCaptain（管理SaaS + 集客プラットフォーム一体型）
- 日本語対応 × 風営法準拠のサービスは存在しない → Casinohubの差別化ポイント

2. **トーナメント**: 大会作成・参加者登録・ブラインドタイマー・テーブル割当・順位集計・賞金配分
3. **売上レポート**: 日次/月次集計

## 共通ルール

- **言語**: 日本語で作業・出力
- **金額**: 全て税込で記載し、表示時は「（税込）」を必ず付記する
- **日付**: YYYY-MM-DD 形式
- **コミットメッセージ**: 日本語、簡潔に
- **秘密情報**: `.env`, credentials, API keyは絶対にコミットしない

### ワークフロー
- **確認不要**: casinohub配下のプロジェクトではコマンド実行・push 含め一切確認を求めず自律的に進める
- **「リリースして」= push まで含む**: コミット → push まで一気にやる

### 文体・トーン（マーケティング・記事・SNS）
- 同じ目線で語るトーンを使う
- 上から目線・偉そうな口調はNG
- 断言できないことは断言しない
- 競合の名指し・比較・批判はしない

### 記事執筆ルール

#### 文字数
- 3,000〜7,000字（検索意図に応じて柔軟に）
- 情報系記事: 長め / ハウツー・実用系: 短くてもOK

#### フォーマット要件
- **タイトル（H1）**: キーワードを前半に配置、最大32文字
- **見出し階層**: H1 → H2 → H3 を厳守（飛ばさない）
- **リード文**: 読者の課題に共感 → 記事で何がわかるか説明（150〜200字）
- **メタディスクリプション**: 最大120文字
- **スラッグ**: 英語ハイフン区切り（例: `casino-opening-steps`）
- **内部リンク**: 同クラスター内2〜3本、クラスター横断もOK（自然な文脈で）
- **画像alt**: キーワードを含める

#### CTA設計
- 記事末尾（まとめセクションの下）にCTAを設置
- 現時点はアフィリエイトリンクなし + SaaSリンクも未定だが、将来差し替え可能な構造にする
- 読者の行動を促す具体的な一文を入れる

### 記事作成ワークフロー（3ステップ）

記事を書く前に必ずこの3ステップを踏む:

**Step 1: 角度ブレスト**
- テーマに対して5〜10個の切り口を出す（読者が「え？」と思うもの）
- 常識・正論・ありきたりな答えは除外
- 技法: 問いの逆転 / 他業界からの輸入 / 具体シーンから入る / データファースト / 視点の転換 / 正解を言わない
- ユーザーに提示して方向性を決める

**Step 2: 具体 ↔ 抽象の設計**
- **具体（入口）**: 読者が自分の経験から「あるある」と思える場面描写
- **抽象（本質）**: より深い構造・問いで「なるほど」と思わせる
- 具体 → 抽象の順で書く（逆にしない）

**Step 3: 執筆**
- Step 1-2で方向が決まってから書き始める
- 毎回同じ構成（問題提起→なぜダメか→代替案→まとめ）にしない
- 結論が構成から予測できない記事にする

### ライティングテクニック
- **断言する**: 「〜と言われています」「〜かもしれません」を減らす。根拠があることは言い切る
- **具体シーンを描く**: 「金曜の22時、トーナメント参加者が30人を超えて、手書きの組み合わせ表では追いつかない」のような場面
- **書き手の意見を入れる**: 「正直、これはもったいない」「ここが一番見落とされがちだ」
- **リズムをつくる**: 長文＋短文、ダッシュ、問いかけ、体言止めを織り交ぜる
- **読者に直接語りかける**: 「あなたの店のトーナメント、参加者は増えていますか？」
- **同テーマの表現変化（必須）**: 複数記事で同じトピックに触れる場合、毎回フォーマットを変える（箇条書き↔散文、俯瞰↔具体シーン、三人称↔一人称等）。読者が「前にも同じ文章を読んだ」と感じたらNG

### 禁止コンテンツ
- 虚偽・誇張した表現
- 競合サービス・他店舗の名指し比較・批判
- 法的に不正確な記述（風営法・賭博罪の境界等は特に正確に）
- 他サイトのコピー・リライト
- 「いかがでしたか」等の定型フレーズ
- 上から目線の「教えてあげる」口調
- 換金を示唆・推奨する表現（風営法違反）

### セルフレビュー工程（必須）

全ての成果物を提出する前に、「ユーザーがさらに優秀になったバージョン」を模倣して一度レビュー・ブラッシュアップを行ってから提出すること。

#### セルフレビューチェックリスト（記事）
- [ ] タイトル32文字以内、キーワードが前半にあるか
- [ ] H1→H2→H3の階層が守られているか
- [ ] リード文で読者の課題に共感し、記事の内容を予告しているか
- [ ] メタディスクリプションが120文字以内か
- [ ] 断言できないことを断言していないか
- [ ] 具体シーンが最低1つあるか
- [ ] 同テーマの既存記事と表現が被っていないか
- [ ] 内部リンクが適切に設置されているか
- [ ] CTAが記事末尾にあるか
- [ ] 風営法・賭博罪に関する記述が法的に正確か
- [ ] 換金を示唆する表現がないか
- [ ] 数値データの出典が確認済みか

### Ralph Loop開発フロー（PRD駆動の自律開発）

1. **PRD作成**: `PRD.md` にタスク一覧を作成
2. **PRD深掘り**: 各タスクを実装・テスト・検証・エッジケースまで含めたサブタスクに分解
3. **Ralph Loop実行**: PRDのタスクを上から順に消化。全タスクが `[x]` + 検証PASSまで止まらない
4. **完了タスクは即座に `[x]` 更新** → コミット → 次へ

### 進捗レポート出力（必須）

長時間作業中は、ユーザーが非同期で確認できるように以下のフォーマットで出力する:

**タスク完了時:**
```
[完了] Phase X > X.X タスク名
- 実装内容: 概要
- 検証結果: テスト結果 or ビルド結果
- 確認URL: （デプロイがある場合）
- 次のタスク: X.X タスク名
```

**ブロッカー発生時:**
```
[ブロック] Phase X > X.X タスク名
- 原因: 詳細
- 必要なアクション: ユーザーが何をすればいいか
- スキップして次のタスクに進みます
```

**Phase完了時:**
```
[Phase完了] Phase X: フェーズ名
- 完了: X/X タスク
- ブロック: Xタスク（内容）
- 確認URL: （あれば）
- 次のアクション: Phase X に進む or ブロッカー解消？
```

### 競合ツール状況
- チップ管理: Excel/紙
- POS: Airレジ・スマレジ等の汎用POSを転用
- トーナメント管理: 海外製PC用ソフト（Poker Tournament Manager等）
- 会員管理: LINE公式 + Google Sheets
- **統合管理ツールは市場に存在しない → Casinohubの最大の参入余地**

### 現場の主なペイン
- トーナメント集計のExcelミスでクレーム（順位・賞金配分）
- 参加者受付・チェックインがLINE+紙で混乱
- テーブル割当を頭の中で計算、大会が大きくなるとパンク
- ディーラー採用難（育成に3〜6ヶ月）
- 深夜営業規制による収益機会の損失
- チップ棚卸しは手間だがチップ原価が低いため厳密管理は不要（コスト > ロス）
